# Neighbor-Know: Zero-Knowledge Authentication Architecture

**Objective:** To establish a peer-to-peer marketplace where users can mathematically prove they are active students at the University of Southern California (`@usc.edu`) and maintain a persistent identity, without their personal identifiable information (PII) ever reaching the application's backend server.

---

## Phase 1: Token Acquisition (The Implicit Flow)

Standard OAuth 2.0 relies on the backend to securely exchange an Authorization Code for a token. Because the Neighbor-Know backend operates in a zero-knowledge trust model, it cannot be responsible for fetching the token.

1. **Initiation:** The client frontend utilizes the Google Identity Services (GIS) JavaScript library to initiate the **Implicit Flow** (`response_type=id_token`).
2. **Domain Verification:** Instead of a `client_secret`, Google verifies the request via **Authorized JavaScript Origins** configured in the Google Cloud Console.
3. **Delivery:** Upon user consent, Google returns the signed JSON Web Token (JWT) directly to the client's browser memory.

---

## Phase 2: The Zero-Knowledge Circuit (Client-Side Proving)

Once the browser has the JWT, it loads a WebAssembly (WASM) compiled zero-knowledge circuit (written in Circom) to evaluate the token.

### Circuit Inputs

| Input Type | Data | Description |
| --- | --- | --- |
| **Private** | JWT Payload | The full, unencrypted JWT containing the user's `email`, `sub` (Google ID), `iat`, and `exp`. |
| **Private** | JWT Signature | The RSA-2048 signature attached to the token. |
| **Public** | Google RS256 Key | The public key hash used to verify the signature. |
| **Public** | Domain String | The target string: `@usc.edu`. |
| **Public** | App Pepper | A high-entropy constant unique to Neighbor-Know. |
| **Public** | Current Time | The Unix timestamp generated by the client browser. |

### Circuit Constraints

Inside the circuit, the following mathematical constraints must evaluate to `True` for the proof to compile:

1. **Signature Authentication:** BigInt arithmetic is used to process 2048-bit RSA keys within the prime field, ensuring the token is a genuine Google artifact.
2. **Domain Match:** A Deterministic Finite Automaton (DFA) parses the `email` claim and asserts that the string suffix matches `@usc.edu`.
3. **Audience Check:** Asserts that the `aud` claim matches the Neighbor-Know Google Client ID to prevent token replay from other applications.

---

## Phase 3: Anonymous Identity Generation (ZK Nullifiers)

To allow users to build reputation and history without exposing their identity, the circuit generates a deterministic pseudonym.

1. **Extraction:** The circuit isolates the `sub` claim (the immutable 21-digit Google User ID).
2. **Hashing:** The `sub` and the public `pepper` are passed through a ZK-friendly hash function (Poseidon).
3. **Output:** The circuit outputs the **nullifier** as a public variable. Because Poseidon is deterministic and collision-resistant, the same student will generate the exact same nullifier on every login, ensuring persistence across sessions.

---

## Phase 4: Time and Session Management

Math does not have a clock, so time constraints must be handled explicitly to prevent the reuse of expired JWTs.

1. **In-Circuit Time Check:** The circuit accepts the client's current Unix timestamp as a public input and mathematically asserts that it is strictly less than the token's `exp` claim.
2. **Backend Time Verification:** When the backend receives the proof, it verifies that the client's public timestamp matches the server's current clock (within a ~2-minute network latency window).
3. **Sliding Application Sessions:** Once the backend verifies the proof, the 1-hour Google JWT lifespan is discarded. The backend issues a secure HTTP-only session cookie tied to the nullifier with a sliding expiration (e.g., 7 days of inactivity), managed entirely by the Neighbor-Know database.

---

## Phase 5: Backend Verification & Trust Model

The backend transitions from a model of network trust (trusting a TLS connection to Google) to **cryptographic trust** (trusting the math).

1. **The Payload:** The browser sends only the zk-SNARK proof and the public inputs/outputs (the nullifier, current time, and Google's public key hash). The `email` and JWT are destroyed client-side.
2. **Verification:** The backend runs the fast, constant-time verification algorithm against the circuit's verification key.
3. **Security Guarantee:** Because it is computationally infeasible to forge an RSA-2048 signature, an attacker cannot generate a valid proof for a fake payload. If the proof is valid, the backend has 100% certainty the user is a verified USC student.

---

## Phase 6: Engineering Feasibility & Performance

The primary engineering bottleneck in this architecture is the client-side proof generation time.

Evaluating RSA-2048 signatures inside a `bn128` prime field requires millions of constraints. On typical student hardware—such as a late-2022 MacBook Pro with standard performance throttling—running the standard `snarkjs` prover in the browser can take **10 to 15 seconds**. Optimizations (e.g., proof aggregation, alternative curves, or pre-computation where possible) may be necessary for acceptable UX.

## Relation to Current Implementation

The codebase currently uses a **Semaphore-based** flow: the backend verifies the Google ID token (e.g. `hd=usc.edu`) at enrollment and stores identity commitments in a group; the client proves group membership with a Semaphore proof and derives a stable pseudonym from the commitment. The architecture above describes a **stronger** zero-knowledge design where the backend never sees the JWT and the circuit itself proves “I hold a valid @usc.edu Google JWT” and outputs a deterministic nullifier. See [cross-session-identity.md](./cross-session-identity.md) for how stable pseudonyms are used for listings and chats once the backend trusts the nullifier.
